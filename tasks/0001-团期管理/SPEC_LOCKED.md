# 锁定规格书 — 0001 团期管理

> 基于 tasks/0001-团期管理/CONTEXT.md 生成，经 auto-dev-loop Step1 格式化

## 1. 🎯 核心目标 (Primary Goal)

- **用户故事（管理员）：** 作为后台管理员，我想要每周创建/编辑/取消团期，以便客户可以提前报名参团
- **用户故事（小程序用户）：** 作为小程序用户，我想要按周查看团期列表和剩余名额，以便选择合适的团期报名
- **核心价值：** 实现团期的全生命周期管理（创建→编辑→取消），支撑预约业务的核心数据基础

## 2. 🗺️ 系统范围与边界 (System Scope & Boundaries)

### 包含模块/功能点
- 后台管理员 CRUD 团期（创建、编辑、取消）
- 小程序端按周查询团期列表
- 团期状态机管理（报名中 → 已满 → 已出发 → 已完成）
- 同一线路同一天去重校验
- 团期日期限制（仅周六/周日）
- 名额满自动标记"已满"

### 输入/输出 (I/O) 规范
- **输入：** 管理员通过后台 API 传入团期信息（线路ID、日期、最大名额、领队信息）
- **输出：** JSON 格式的团期数据（含状态、剩余名额、线路关联信息）

### 明确非目标 (Non-Goals)
- ❌ 微信消息通知（V2 实现）
- ❌ 团期自动创建（每周自动生成）
- ❌ 领队排班管理
- ❌ 团期导出/报表功能

## 3. ✅ 验收标准 (Acceptance Criteria)

- **AC-01:** When 管理员提交包含线路ID、出行日期、最大名额、领队信息的创建请求, the system shall 创建团期并返回 200 + 团期详情
- **AC-02:** When 管理员提交已存在团期的编辑请求（修改名额/领队）, the system shall 更新团期并返回 200
- **AC-03:** When 管理员取消一个有已支付订单的团期, the system shall 标记团期为"已取消"并返回 200（退款流程由 Task 0002 处理）
- **AC-04:** When 小程序用户请求按周查询团期列表（传入 week 参数）, the system shall 返回该周所有团期，包含线路名、日期、已报名人数、剩余名额
- **AC-05:** When 团期已报名人数 >= 最大名额, the system shall 自动将状态标记为"已满"，且该团期不可再被预约
- **AC-06:** When 管理员创建与已有团期相同线路+相同日期的团期, the system shall 返回错误码 400 + "该线路该日期已有团期"

## 4. ⛓️ 关键约束与假设 (Constraints & Assumptions)

### 技术约束
- Go + Gin + GORM + MySQL 8.0
- 遵循四层架构：Handler → Service → Repository → Model
- 团期状态机：报名中(enrolling) → 已满(full) → 已出发(departed) → 已完成(completed)，支持直接取消(cancelled)

### 业务约束
- 团期日期只能是周六或周日
- 创建团期时自动校验线路是否存在且已上架
- 已报名人数通过 orders 表实时统计（不冗余存储）

### 核心假设
- routes 表已有初始数据（4 条线路）
- schedules 表 DDL 已在 migrations/001_init.sql 中定义
- 管理员认证通过 JWT 中间件实现（已有 middleware/auth.go）
