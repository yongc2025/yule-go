## 一、双版本战略说明

本项目采用 **双版本并行** 策略，分别服务于不同阶段的目标：

```
                    时间线
                      │
    ┌─────────────────┼─────────────────┐
    │                 │                 │
    ▼                 ▼                 ▼
 cloud-mvp          yule-go           合并
 快速验证           正式产品          数据迁移
 (2-3周)           (6-8周)          (1周)
    │                 │                 │
    └──→ 验证商业模式 ─┘                 │
         收集用户反馈 ──────────────────┘
         测试定价策略
```

| 维度 | cloud-mvp（云开发 MVP） | yule-go（自研版本） |
|:-----|:------------------------|:-------------------|
| **目标** | 快速验证商业模式 | 正式产品，长期维护 |
| **技术栈** | 微信云函数 + 云数据库 | Go (Gin) + MySQL + Redis |
| **部署** | 微信云开发（零运维） | Docker + 云服务器 |
| **管理后台** | 小程序内嵌管理页 | Vue3 + Element Plus 独立 PC 端 |
| **开发周期** | 2-3 周 | 6-8 周 |
| **灵活性** | 受限于云开发能力 | 完全自主可控 |
| **适用阶段** | 0→1 验证 | 1→N 正式运营 |

**核心逻辑**：cloud-mvp 用最短时间上线，验证"周末垂钓出行"这个商业模式是否成立；yule-go 同步开发正式版本，验证成功后用户和数据可迁移。

## 七、风险与缓解措施

### 7.1 技术风险

| 风险 | 影响 | 概率 | 缓解措施 |
|:-----|:-----|:-----|:---------|
| 微信支付对接复杂 | 支付功能延期 | 中 | 提前研读文档，准备沙箱环境，预留 2 天缓冲 |

### 7.2 业务风险

| 风险 | 影响 | 概率 | 缓解措施 |
|:-----|:-----|:-----|:---------|
| 用户不接受小程序预约 | 转化率低 | 中 | 保留电话预约通道，小程序作为辅助 |

| 风险 | 影响 | 概率 | 缓解措施 |

| 需求变更频繁 | 开发周期超预期 | 高 | 需求冻结机制，变更走正式流程 |
