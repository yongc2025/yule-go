# 需求差距全面分析 — yule-go

> **专家团队**：产品经理 × 钓场运营专家 × 旅游服务专家 × 支付安全专家 × 系统架构师
> **日期**：2026-05-06
> **版本**：v1.0
> **目标**：对照商业计划书需求，逐模块分析现有实现的遗漏与不足

---

## 一、分析方法论

```
商业计划书原始需求
       ↓
现有代码实现（42 Go + 17 前端文件）
       ↓
逐场景走查 → 标记覆盖/部分/缺失
       ↓
按用户旅程 + 运营旅程 + 系统职责三维交叉验证
```

**覆盖度判定标准**：
- ✅ **已覆盖**：代码已实现，可直接使用
- ⚠️ **部分覆盖**：有骨架但缺关键逻辑
- ❌ **完全缺失**：无任何代码支撑
- 🔴 **阻塞性缺失**：不实现则核心流程无法跑通

## 三、逐模块详细分析

### 3.1 用户身份模块

#### 需求原文
> 小程序操作不超过 3 步完成预约；微信支付 HTTPS

#### 现状
- ❌ **微信登录未实现** — `model/user.go` 有 User 模型，但无 `handler/user.go`、无微信登录接口
- ❌ **小程序授权流程缺失** — 无 `wx.login()` → 后端 `code2session` → 返回 JWT 的完整链路
- ❌ **用户注册/创建未实现** — 用户表存在但无创建逻辑
- ❌ **JWT 签发给小程序用户** — 现有 JWT 仅管理员使用，小程序用户无 token

#### 需要实现
```
小程序端登录流程：
1. 小程序调用 wx.login() 获取 code
2. 后端 /api/v1/auth/wx-login 接收 code
3. 后端调用微信 code2session API 换取 openid + session_key
4. 查询/创建用户记录
5. 签发 JWT token（区分 user/admin）
6. 返回 token + 用户基本信息

额外考虑：
- unionid 获取（关联公众号/开放平台时需要）
- 手机号获取（微信 getPhoneNumber 组件）
- 用户信息更新（昵称/头像授权）
- token 刷新机制
```

#### 遗漏场景
| 场景 | 说明 |
|:-----|:-----|
| 首次登录 | 新用户无 invite_code，需要自动生成 |
| 手机号绑定 | 需求提到手机号，但微信获取手机号需用户授权 |
| 用户注销 | GDPR 合规，需要用户注销入口 |
| 多设备登录 | 同一用户多设备 token 管理 |

### 3.2 支付模块

#### 需求原文
> 选日期 → 选线路 → 填人数 → 微信支付 → 生成订单

#### 现状
- ❌ **微信支付完全未实现** — 仅有 `WxPaymentParams` 结构体定义
- ❌ **支付统一下单接口缺失** — 无调用微信支付 API v3 的代码
- ⚠️ **支付回调已有骨架** — `PaymentCallback` 仅更新状态，签名验证是简单 SHA256 而非微信标准签名
- ❌ **退款接口缺失** — 无微信退款 API 调用
- ❌ **支付结果查询缺失** — 无主动查询支付结果的逻辑

#### 需要实现
```
支付完整链路：
1. 订单创建 → 生成预支付交易会话（调用微信统一下单 API）
2. 返回支付参数给小程序 → 小程序调用 wx.requestPayment()
3. 支付成功 → 微信异步通知后端（需验签）
4. 后端更新订单状态 → 触发后续流程
5. 支付超时 → 自动关闭（已有定时任务，但需同步关闭微信侧订单）

退款完整链路：
1. 用户申请退款 → 进入退款审核队列
2. 管理员审核 → 通过/驳回
3. 通过 → 调用微信退款 API
4. 微信退款回调 → 更新退款状态
5. 退款失败 → 重试 + 人工兜底
```

#### 遗漏场景
| 场景 | 说明 |
|:-----|:-----|
| 部分退款 | 用户租赁了装备但只想退团费 |
| 退款到原支付方式 | 微信退款需原路退回 |
| 支付金额不一致 | 前端计算 vs 后端计算 vs 微信实付 的三方校验 |
| 重复支付 | 用户重复点击支付按钮 |
| 支付掉单 | 微信回调丢失，需主动查询补单 |
| 余额+微信混合支付 | 余额抵扣一部分，微信支付剩余部分 |

### 3.3 钓场/钓位模块

#### 需求原文
> 依托现有渔具实体店，打造"零售 + 短途旅行"双盈利模式

#### 现状
- ❌ **钓场表不存在** — 无 `fishing_spots` 表
- ❌ **钓位体系不存在** — 无 `spot_grades`、`fishing_positions` 表
- ❌ **团期钓位库存不存在** — 无 `schedule_positions` 表
- ❌ **线路-钓场关联缺失** — `routes` 表无 `fishing_spot_id` 字段
- ❌ **钓场管理后台缺失** — 无钓场 CRUD 页面

#### 业务影响分析

**现状的隐含假设**：所有线路去同一个地方、所有名额等价、不需要知道去哪钓。

**实际业务**：
- 一个渔具店通常对接 **多个钓场**（不同水库、不同鱼塘）
- 同一钓场有 **不同等级的钓位**（VIP 近水位、普通位、远岸位）
- 不同钓位 **价格差异显著**（VIP 位可能是普通位的 2-3 倍）
- 用户预约时需要 **选择具体钓位**，而不是到了再分配
- 用户需要 **查找附近钓场**，按距离筛选

#### 数据模型设计（详见 `missing-entities-analysis.md`）

核心新增 5 张表：
1. `fishing_spots` — 钓场
2. `spot_grades` — 钓位等级
3. `fishing_positions` — 钓位
4. `schedule_positions` — 团期钓位库存
5. `route_spots` — 线路-钓场关联（或直接在 routes 加 fishing_spot_id）

#### 遗漏场景
| 场景 | 说明 |
|:-----|:-----|
| 钓场营业时间限制 | 用户不能预约钓场非营业时段的团期 |
| 钓位维护 | 某钓位损坏需临时禁用 |
| 钓位热力图 | 运营需要知道哪些钓位最受欢迎 |
| 多钓场线路 | 一条线路可能途经多个钓场 |
| 钓位价格动态调整 | 旺季/淡季不同价格 |
| 钓场容量限制 | 钓场同时接待人数有上限 |

### 3.4 核销模块

#### 需求原文
> 出行保险、全程陪同、领队服务

#### 现状
- ❌ **核销记录表不存在** — 无 `checkins` 表
- ❌ **核销码生成缺失** — 下单后不生成核销码
- ❌ **核销二维码展示缺失** — 小程序无法展示核销码
- ❌ **扫码核销功能缺失** — 管理后台无核销页面
- ❌ **GPS 辅助核销缺失** — 无法校验用户是否在钓场范围

#### 核销完整流程设计

```
用户端：
1. 支付成功 → 自动生成核销码（6-8位数字）+ 二维码
2. "我的订单" → 展示核销二维码
3. 出行当天 → 打开小程序 → 出示二维码
4. 核销完成 → 订单状态更新

管理端/领队端：
1. 打开核销页面 → 选择本期团期
2. 扫描用户二维码 / 手动输入核销码
3. 确认核销 → 实时更新到场统计
4. 全部核销完成 → 标记团期结束
```

#### 遗漏场景
| 场景 | 说明 |
|:-----|:-----|
| 核销码过期 | 团期结束后核销码自动失效 |
| 重复核销 | 同一订单多次核销需幂等 |
| 代他人核销 | 一人帮全家核销（亲子团场景） |
| 核销后退款 | 已核销的订单能否退款？ |
| 离线核销 | 钓场可能没信号，需要离线方案 |
| 核销数据统计 | 到场率、迟到率等 |

### 3.5 订单生命周期模块

#### 需求原文
> 我的订单：历史订单、待出行、已出行、退款

#### 现状
- ⚠️ **状态机过于简单** — 仅有 6 个状态（待支付/已确认/已出行/已完成/已取消/已退款）
- ❌ **缺少"退款中"状态** — 用户申请退款到退款完成之间的中间态
- ❌ **缺少"待核销"状态** — 到场后等待核销
- ❌ **取消退款规则缺失** — 无基于时间的退款金额计算
- ❌ **自动完成缺失** — 团期结束后订单不会自动流转到"已完成"
- ❌ **订单过期未支付处理** — 已有定时任务，但未释放钓位库存

#### 建议状态机（8 状态）

```
PENDING(待支付) → PAID(已支付) → EN_ROUTE(出行中) → CHECKIN(待核销) → COMPLETED(已完成)
       ↓                ↓                                                         ↑
  CANCELLED(已取消)  REFUNDING(退款中) → REFUNDED(已退款)                    自动/手动完成
```

#### 退款规则设计

| 取消时间 | 退款比例 | 说明 |
|:---------|:---------|:-----|
| 出行前 72 小时+ | 100% 全退 | 提前充分 |
| 出行前 48-72 小时 | 80% | 扣 20% 作为占位费 |
| 出行前 24-48 小时 | 50% | 半退 |
| 出行前 24 小时内 | 不退 | 已产生成本 |
| 天气/不可抗力 | 100% 全退 | 非用户原因 |

#### 遗漏场景
| 场景 | 说明 |
|:-----|:-----|
| 订单改期 | 用户想换到下周，不是取消重订 |
| 转让订单 | 用户把名额转给朋友 |
| 部分退款 | 3 人团退 1 人 |
| 退款审核驳回 | 管理员驳回退款申请 |
| 订单申诉 | 用户对退款结果不满 |
| 订单备注（管理员） | 管理员给订单加内部备注 |

### 3.7 老带新裂变模块

#### 需求原文
> 老客户返现 20 元 / 新客户首单立减 15 元 / 2人同行每人立减 10 元 / 3人及以上每人立减 15 元 + 赠送鱼饵礼包

#### 现状
- ⚠️ **基础裂变已实现** — 邀请码生成、新客立减 15、老客返现 20
- ❌ **同行优惠未实现** — "2人同行/3人及以上"梯度折扣未实现
- ❌ **鱼饵礼包未实现** — 3人及以上的赠品逻辑缺失
- ❌ **裂变数据看板缺失** — 无法查看邀请转化率

#### 同行优惠规则详解

```
场景 A：2 人同行（其中 1 人是老用户邀请）
  → 老用户：返现 20 + 同行立减 10 = 共减 30
  → 新用户：首单立减 15 + 同行立减 10 = 共减 25

场景 B：3 人同行（其中 1 人是老用户邀请）
  → 老用户：返现 20 + 同行立减 15 + 鱼饵礼包 = 共减 35 + 赠品
  → 新用户 A：首单立减 15 + 同行立减 15 = 共减 30 + 赠品
  → 新用户 B：同行立减 15 = 减 15 + 赠品

场景 C：3 人同行（无邀请关系，纯同行）
  → 每人立减 15 + 鱼饵礼包
```

#### 遗漏场景
| 场景 | 说明 |
|:-----|:-----|
| 同行关系建立 | 如何判定"同行"？同一订单？同一团期？ |
| 同行优惠与会员折扣叠加 | 两个优惠能否同时使用？ |
| 邀请码分享路径 | 小程序分享 → 带参数二维码 → 扫码注册 |
| 裂变奖励发放方式 | 到余额？到优惠券？下次自动抵扣？ |
| 防刷机制 | 同一人反复注册新号薅羊毛 |

## 八、结论

### 现有实现覆盖率评估

| 模块 | 覆盖率 | 说明 |
|:-----|:-------|:-----|
| 数据模型 | 40% | 基础表有，钓场/钓位/核销缺失 |
| 后端 API | 35% | 核心链路断在支付和钓场 |
| 管理后台 | 25% | 仅团期管理可用 |
| 小程序端 | 30% | 团期/订单/预约有骨架，首页/钓场/核销缺失 |
| 定时任务 | 15% | 仅超时取消 |

### 核心结论

**当前系统可以"看"但不能"用"**：
- 管理员可以创建团期，但无法管理钓场和钓位
- 用户可以浏览团期和预约，但无法支付
- 用户可以看订单，但没有核销码
- 裂变基础功能有，但同行优惠未实现

**要达到 MVP 可用状态，需要完成约 24 天的开发工作**，重点是：
1. 微信登录 + 支付（5 天）
2. 钓场/钓位体系（7 天）
3. 核销 + 退款（5 天）
4. 首页 + 线路管理 + 状态机（7 天）
