# AI 软件开发规则 (AI Software Development Rules)

> 适用于 AI 辅助编码、代码审查、项目理解等场景。
> 版本: v1.0 | 2026-05-03

---

## 一、代码阅读规则

### 1.1 分层阅读，先广后深

```
第一步: 文件树 → 理解项目结构（目录、文件名、层级关系）
第二步: README / 文档 → 理解项目定位、功能清单、技术栈
第三步: 入口文件 → run.py / main.py / app.py → 理解启动链路
第四步: 核心模块 → engine / core / service → 理解业务逻辑
第五步: 配置 / 常量 → config.py / settings → 理解可调参数
```

**禁止行为：**
- ❌ 不看结构直接读单个文件
- ❌ 跳过 README 直接读代码
- ❌ 只读一个文件就下结论

### 1.2 文件读取策略

| 文件类型 | 策略 | maxChars 建议 |
|---------|------|--------------|
| README / 文档 | 全量读取 | 不设限（或 50000） |
| 配置文件 (config/settings) | 全量读取 | 不设限 |
| 入口文件 (run/main/app) | 全量读取 | 不设限 |
| 核心业务逻辑 | 全量读取 | 不设限 |
| 测试文件 | 先读结构，按需深入 | 20000 |
| 模板/静态文件 | 按需读取 | 15000 |
| 超大文件 (>500行) | 分段读取（offset + limit） | 分段 3000 |

**原则：代码文件尽量不限制 maxChars，宁可多读不要漏读。**

### 1.3 理解深度标准

阅读一个模块后，必须能回答：

- [ ] 这个模块的**输入**是什么？（参数、数据来源）
- [ ] 这个模块的**输出**是什么？（返回值、副作用）
- [ ] 核心**流程**是什么？（主干逻辑 3-5 步）
- [ ] 有哪些**边界条件**和**错误处理**？
- [ ] 和其他模块的**依赖关系**是什么？

## 二、代码编写规则

### 2.1 先理解再动手

- 修改代码前，必须先读懂**当前文件的完整逻辑**
- 修改代码前，必须理解**调用链**（谁调用它，它调用谁）
- 不确定影响范围时，先搜索所有引用

### 2.2 最小改动原则

- 只改需要改的，不做"顺便优化"
- 不要重构你不完全理解的代码
- 一次只解决一个问题（除非用户明确要求批量修改）

### 2.3 代码风格

- **遵循项目已有风格**（缩进、命名、注释语言），不要引入新风格
- 变量命名清晰，拒绝 `a`, `b`, `tmp`, `data2` 这类命名
- 注释解释"为什么"，而不是"做什么"（代码本身应该说明"做什么"）
- 错误信息要对用户友好，不要暴露内部实现细节

### 2.4 错误处理

```python
# ✅ 好的做法：具体、可操作、用户友好
except httpx.ConnectError:
    return {"error": f"网络连接失败，请检查代理端口 ({PROXY_URL}) 是否开启"}

# ❌ 坏的做法：吞掉异常或暴露内部信息
except Exception as e:
    return {"error": str(e)}
```

- 捕获具体异常，不要裸 `except Exception`
- 用户侧错误信息要**可操作**（告诉用户该怎么做）
- 开发侧日志要**可追踪**（包含足够的上下文）

### 2.5 异步代码规范

- 使用 `async/await`，不要混用同步阻塞调用
- 并发操作使用 `asyncio.gather`，不要串行等待
- 超时操作使用 `asyncio.wait_for`，设置合理的 timeout
- WebSocket / 长连接必须有重连机制

## 三、项目理解规则

### 3.1 理解一个项目的标准流程

```
1. 定位 → 这个项目解决什么问题？面向什么用户？
2. 结构 → 目录怎么划分的？每层的职责是什么？
3. 数据流 → 数据从哪来？经过什么处理？到哪去？
4. 核心链路 → 主流程是什么？（用 3-5 步概括）
5. 配置项 → 哪些参数可调？默认值是什么？
6. 边界 → 什么不支持？什么没做？
```

### 3.2 讲解规范

给用户讲解项目时：

- **先总后分**：先用一段话概括，再展开细节
- **用表格**梳理模块/功能/API，不要用大段文字
- **标注优先级**：哪些是核心功能，哪些是辅助功能
- **指出亮点和风险**：不只是描述，要有观点

## 四、文件操作规则

### 4.1 读取规则

- 读取前先确认文件大小（通过文件树 API）
- 小文件（<10KB）：直接全量读取
- 中等文件（10-50KB）：全量读取，不设 maxChars
- 大文件（>50KB）：分段读取，每段 3000 行以内
- **绝不因为"可能太长"就随意截断**

### 4.2 写入规则

- 写入新文件前，确认目录存在
- 覆盖写入前，确认用户意图（或保留备份）
- 写入后验证：重新读取确认内容正确

### 4.3 编辑规则

- 使用精确替换（`oldText` → `newText`），不用全文覆盖
- `oldText` 必须是文件中**精确存在**的文本（含空格、缩进）
- 替换后检查上下文是否仍然连贯

## 五、沟通规则

### 5.1 回复风格

- **解决问题优先**：少说"我来帮你"，直接做
- **有观点**：可以不同意，可以有偏好，可以指出问题
- **结构化**：用标题、表格、代码块组织信息，不要大段纯文本
- **适度简洁**：简单问题一句话回答，复杂问题再展开

### 5.2 不确定时

- 说"我不确定"比瞎编好
- 先搜索/验证再回答，不要凭记忆猜测
- 如果代码有歧义，列出多种可能的理解，请用户确认

### 5.3 主动性

- 发现明显 bug → 主动指出
- 发现安全隐患 → 主动提醒
- 发现更好的实现方式 → 建议但不强改
- 用户没问但很重要的事情 → 顺带提一句

## 六、安全规则（开发场景）

- ❌ 不在代码中硬编码 API Key / Secret / Token
- ❌ 不把 `.env` / 配置文件内容输出到聊天
- ❌ 不在用户可见的日志/错误信息中暴露内部路径、堆栈、密钥
- ✅ 加密存储敏感信息
- ✅ 错误信息脱敏后再展示
- ✅ 代码审查时检查是否有敏感信息泄露

---

## 七、规则自检清单

每次完成一个开发任务后，自问：

- [ ] 我是否完整理解了相关代码后再动手？
- [ ] 我的改动是否最小化？有没有顺便改了不该改的？
- [ ] 错误处理是否完善？用户看到的错误信息是否有用？
- [ ] 我的回复是否结构清晰？用户能否直接行动？
- [ ] 是否有安全隐患？
