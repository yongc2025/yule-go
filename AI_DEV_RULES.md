# AI 开发规则 (AI Development Rules)

> 适用于 AI 辅助编码、代码审查、项目理解等场景。
> 每次写代码前必须阅读本文件。
> 版本: v1.0 | 2026-05-08 | 适配 yule-go 项目

---

## 一、代码阅读规则

### 1.1 分层阅读，先广后深

```
第一步: 文件树 → 理解项目结构（目录、文件名、层级关系）
第二步: AGENTS.md + memory-bank/ → 理解项目定位、架构、当前进度
第三步: 入口文件 → main.go / app.json → 理解启动链路
第四步: 核心模块 → service/ (Go) / cloudfunctions/ (云函数) → 理解业务逻辑
第五步: 共享配置 → shared/config.js → 理解业务规则和可调参数
第六步: 当前任务 → tasks/000X/ → 理解要做什么
```

**禁止行为：**
- ❌ 不看结构直接读单个文件
- ❌ 不读 AGENTS.md 和 memory-bank 就开始改代码
- ❌ 只读一个文件就下结论
- ❌ 不读 shared/config.js 就硬编码业务规则

### 1.2 文件读取策略

| 文件类型 | 策略 | 说明 |
|:---|:---|:---|
| AGENTS.md / 规范文档 | 全量读取 | 每次会话必读 |
| memory-bank/ | 全量读取 | 理解上下文 |
| shared/config.js | 全量读取 | 写云函数前必读 |
| Go 入口 (main.go) | 全量读取 | 理解启动链路 |
| Go 核心逻辑 (service/) | 全量读取 | 理解业务 |
| 云函数 (index.js) | 全量读取 | 理解 action 和逻辑 |
| 测试文件 | 按需读取 | 先读结构，按需深入 |
| 前端页面 (.vue/.wxml) | 全量读取 | 理解 UI 和交互 |
| tasks/ 目录 | 全量读取 | 理解任务上下文 |

### 1.3 理解深度标准

阅读一个模块后，必须能回答：

- [ ] 这个模块的**输入**是什么？（参数、数据来源）
- [ ] 这个模块的**输出**是什么？（返回值、副作用）
- [ ] 核心**流程**是什么？（主干逻辑 3-5 步）
- [ ] 有哪些**边界条件**和**错误处理**？
- [ ] 和其他模块的**依赖关系**是什么？
- [ ] **业务规则**是否从 shared/config.js 读取？（云函数场景）

## 二、代码编写规则

### 2.1 先理解再动手

- 修改代码前，必须先读懂**当前文件的完整逻辑**
- 修改代码前，必须理解**调用链**（谁调用它，它调用谁）
- 修改云函数前，必须先读 `shared/config.js`
- 不确定影响范围时，先搜索所有引用

### 2.2 最小改动原则

- 只改需要改的，不做"顺便优化"
- 不要重构你不完全理解的代码
- 一次只解决一个问题（除非用户明确要求批量修改）
- 改配置时只改 `shared/config.js`，不要在云函数中单独改

### 2.3 代码风格

- **遵循项目已有风格**（缩进、命名、注释语言），不要引入新风格
- Go 用 `gofmt`，JS 用项目现有的格式化风格
- 变量命名清晰，拒绝 `a`, `b`, `tmp`, `data2` 这类命名
- 注释解释"为什么"，而不是"做什么"（代码本身应该说明"做什么"）
- 错误信息要对用户友好，不要暴露内部实现细节

### 2.4 云函数编写规范

```javascript
// ✅ 正确：从 shared/config.js 读取业务规则
const { COUPON_TYPES, REWARDS } = require('../shared/config')

async function grantCoupon(openid, { type, amount }) {
    const typeConfig = COUPON_TYPES[type]
    if (!typeConfig) {
        return { code: -1, message: '无效的券类型' }
    }
    const couponAmount = amount || typeConfig.amount  // 用配置的默认值
    // ...
}

// ❌ 错误：硬编码业务规则
async function grantCoupon(openid, { type, amount }) {
    const couponAmount = amount || 10  // 魔法数字！
    // ...
}
```

### 2.5 错误处理

```javascript
// ✅ 云函数：具体、可操作、用户友好
try {
    const couponRes = await db.collection('coupons').doc(couponId).get()
    // ...
} catch (err) {
    console.error('[orders.calcDiscount] coupon error:', couponId, err)
    return { code: -1, message: '优惠券查询失败，请重试' }
}

// ❌ 吞掉异常
try { ... } catch (e) { }

// ❌ 暴露内部信息
return { code: -1, message: 'coupons.doc(' + couponId + ').get() failed: ' + err.message }
```

## 三、项目理解规则

### 3.1 理解项目的标准流程

```
1. 定位 → 渔具店周末短途旅行数字化运营，面向微信小程序用户
2. 结构 → Go 后端 + 云函数 + uni-app 小程序 + Vue3 管理后台
3. 数据流 → 小程序 → 云函数 → 云数据库(MongoDB) / Go后端 → MySQL
4. 核心链路 → 用户注册 → 充值会员 → 选团期下单 → 支付 → 核销出行
5. 配置项 → shared/config.js（券类型、会员等级、优惠金额）
6. 边界 → 云函数和 Go 后端是两套独立系统，通过数据库共享数据
```

### 3.2 项目双系统架构

本项目有两套并行的后端系统：

| 维度 | Go 后端 | 云函数 |
|:---|:---|:---|
| 数据库 | MySQL | MongoDB (云数据库) |
| 框架 | Gin + GORM | wx-server-sdk |
| 部署 | 服务器 | 微信云开发 |
| 路由 | router/*.go | action switch-case |
| 分层 | Handler → Service → Repository → Model | exports.main → action handler |
| 配置 | config.yaml | shared/config.js |

**重要**：修改业务规则时，要确认改的是哪个系统。云函数的业务规则在 `shared/config.js`。

### 3.3 讲解规范

给用户讲解项目时：

- **先总后分**：先用一段话概括，再展开细节
- **用表格**梳理模块/功能/API，不要用大段文字
- **标注优先级**：哪些是核心功能，哪些是辅助功能
- **指出亮点和风险**：不只是描述，要有观点

## 四、文件操作规则

### 4.1 读取规则

- 读取前先确认文件大小（通过文件树 API 或 `wc -l`）
- 小文件（<10KB）：直接全量读取
- 大文件（>50KB）：分段读取，每段 3000 行以内
- **绝不因为"可能太长"就随意截断**

### 4.2 写入规则

- 写入新文件前，确认目录存在
- 覆盖写入前，确认用户意图（或保留备份）
- 写入后验证：重新读取确认内容正确
- 云函数修改后，需要提醒用户**重新部署**

### 4.3 编辑规则

- 使用精确替换（`oldText` → `newText`），不用全文覆盖
- `oldText` 必须是文件中**精确存在**的文本（含空格、缩进）
- 替换后检查上下文是否仍然连贯
- 编辑云函数后，必须提醒用户重新部署

## 五、沟通规则

### 5.1 回复风格

- **解决问题优先**：少说"我来帮你"，直接做
- **有观点**：可以不同意，可以有偏好，可以指出问题
- **结构化**：用标题、表格、代码块组织信息，不要大段纯文本
- **适度简洁**：简单问题一句话回答，复杂问题再展开

### 5.2 不确定时

- 说"我不确定"比瞎编好
- 先搜索/验证再回答，不要凭记忆猜测
- 如果代码有歧义，列出多种可能的理解，请用户确认

### 5.3 主动性

- 发现明显 bug → 主动指出
- 发现安全隐患 → 主动提醒
- 发现硬编码的业务规则 → 建议移到 shared/config.js
- 发现云函数间配置不一致 → 立即指出
- 用户没问但很重要的事情 → 顺带提一句

## 六、安全规则

- ❌ 不在代码中硬编码 API Key / Secret / Token / AppSecret
- ❌ 不把 `.env` / 配置文件内容输出到聊天
- ❌ 不在错误信息中暴露集合名、字段名、堆栈
- ❌ 不在日志中输出用户 openid 等敏感信息（开发调试除外）
- ✅ 微信支付密钥通过环境变量 `process.env.SUB_MCH_ID` 读取
- ✅ 代码审查时检查是否有敏感信息泄露

---

## 七、云函数部署提醒规则

每次修改云函数代码后，**必须提醒用户**：

```
⚠️ 云函数修改完成，需要重新部署：
1. 微信开发者工具 → cloudfunctions/ 目录
2. 右键修改的云函数 → "上传并部署：云端安装依赖"
3. 如果修改了 shared/config.js → 所有引用它的云函数都需要重新部署
```

## 八、规则自检清单

每次完成一个开发任务后，自问：

- [ ] 我是否完整理解了相关代码后再动手？
- [ ] 我的改动是否最小化？有没有顺便改了不该改的？
- [ ] 业务规则是否集中在 shared/config.js？（云函数场景）
- [ ] 错误处理是否完善？用户看到的错误信息是否有用？
- [ ] 日期是否用 Date 对象？金额是否精度正确？
- [ ] 我的回复是否结构清晰？用户能否直接行动？
- [ ] 是否有安全隐患？
- [ ] 是否提醒用户重新部署云函数？
