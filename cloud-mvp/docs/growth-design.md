# 渔乐出行 — 获客留客体系设计方案

> **版本**：v1.0
> **日期**：2026-05-08
> **目标**：为 cloud-mvp 设计会员充值、裂变获客、优惠券三大核心模块
> **方法论**：商业计划对齐 + 竞品分析 + 用户行为模型 + 专家评审

---

## 一、商业计划核心提取

### 1.1 业务本质

```
线下渔具店（信任入口）→ 周末短途出行（高频消费）→ 双向盈利（旅行+零售）
```

**关键商业指标**：
| 指标 | 数值 | 来源 |
|:---|:---|:---|
| 客单价 | ¥98 ~ ¥238 | 四大产品定价 |
| 消费频次 | 每月 2~4 次（周末） | 周末固定发团 |
| 目标复购率 | >60% | 商业计划"复购率极高" |
| 获客渠道 | 门店(主) + 私域 + 线上 | 5.1~5.3 |
| 核心人群 | 3 类，年龄跨度 25~60+ | 第三章 |

### 1.2 商业计划中已明确的营销策略

| 策略 | 商业计划描述 | 优先级 |
|:---|:---|:---|
| **会员充值** | 三档(200/500/1000)，渔具折扣+旅行立减+赠品 | 🔴 锁客核心 |
| **老带新裂变** | 老客返¥20/新客首单减¥15 | 🔴 获客核心 |
| **多人同行优惠** | 2人各减¥10，3人+各减¥15+赠鱼饵 | 🟡 拼团 |
| **新人首单立减** | 新注册用户专享 | 🟡 拉新 |
| **四季主题活动** | 春钓/夏营/秋游/冬慢旅 | 🟢 运营 |

### 1.3 关键约束

- **没有商户号** → 充值和裂变奖励不能走真实微信支付，需要模拟/余额体系
- **三类人群差异大** → 钓鱼发烧友(高频高客单) vs 亲子(低频高客单) vs 退休(高频低客单)
- **门店是信任核心** → 所有营销必须强化"门店"概念，不能做成纯线上套路

## 二、竞品分析

### 2.1 竞品矩阵

| 竞品类型 | 代表 | 会员体系 | 裂变方式 | 优惠券 | 我方差异化 |
|:---|:---|:---|:---|:---|:---|
| **本地生活平台** | 美团/大众点评 | 会员卡(¥X/月) | 拼团/砍价 | 满减券/折扣券 | 我们有专属钓点+实体信任 |
| **户外活动平台** | 野友网/探路者 | 积分体系 | 分享返佣 | 早鸟价/团报价 | 我们有装备零售闭环 |
| **旅行社跟团** | 携程/飞猪 | 无 | 拼团 | 优惠码 | 我们是垂钓专属+本地高频 |
| **渔具电商** | 钓鱼之家 | 会员等级 | 分享有礼 | 满减券 | 我们有线下体验+出行服务 |
| **社区团购** | 美团优选 | 无 | 邀请返现 | 新人券 | 我们是体验服务非标品 |

### 2.2 竞品功能拆解

#### 美团/大众点评（本地生活标杆）

**会员体系**：
- 付费会员卡（¥X/月），享专属折扣
- 积分累计，可兑换优惠券
- 会员日专属活动

**裂变**：
- 拼团（2人团/3人团，价格阶梯）
- 分享领红包（分享者+被分享者各得）
- 邀请有礼（邀请注册得优惠券包）

**优惠券**：
- 新人券包（注册即送多张不同面额）
- 满减券（满XX减XX）
- 品类券（指定品类可用）
- 限时券（有效期短，制造紧迫感）

#### 户外活动平台（野友网/探路者俱乐部）

**会员**：
- 年费会员享优先报名+折扣
- 参与次数累计升级

**裂变**：
- 老用户推荐新用户，双方各得余额
- 组队报名享团购价

## 三、用户行为模型分析

### 3.1 三类用户画像

| 维度 | 🎣 钓鱼发烧友 | 👨‍👩‍👧 亲子家庭 | 👴 退休长者 |
|:---|:---|:---|:---|
| **年龄** | 25-60 | 28-45 | 55+ |
| **消费频次** | 高（每周/每两周） | 低（每月1-2次） | 高（每周） |
| **客单价** | ¥128-198 | ¥238 | ¥98 |
| **决策因素** | 钓点质量、领队专业 | 安全、趣味、便捷 | 价格、节奏慢、同伴 |
| **传播意愿** | 高（钓友圈） | 中（家长群） | 高（老同事群） |
| **价格敏感度** | 中 | 低 | 高 |
| **充值意愿** | 高（高频复购） | 低（频次不够） | 中（看优惠力度） |

### 3.2 用户生命周期模型

```
新用户 ──首单立减──→ 首次体验 ──满意──→ 复购用户 ──充值──→ 会员 ──裂变──→ 推荐者
  │                    │                    │                    │
  └─流失               └─差评/流失          └─沉睡               └─持续活跃
```

**关键转化节点**：
1. **新→首单**：需要降低决策门槛（首单立减、新人券）
2. **首单→复购**：需要体验驱动 + 复购激励（积分、会员折扣）
3. **复购→充值**：需要算账逻辑（"充500省XX元"）
4. **会员→推荐**：需要双向奖励（推荐返现 + 新人立减）

### 3.3 RFM 分层策略

| 用户分层 | 特征 | 策略 |
|:---|:---|:---|
| **高价值** | 最近消费近 + 频次高 + 客单高 | 会员升级邀请 + 专属活动 |
| **潜力** | 最近消费近 + 频次低 | 复购券 + 拼团邀请 |
| **沉睡** | 最近消费远 + 曾经高频 | 唤醒券 + 新活动推送 |
| **流失** | 最近消费远 + 颞次低 | 大额挽回券 |

## 四、设计方案

### 4.1 设计原则

1. **简单优先**：MVP 阶段不做复杂积分系统，用余额+优惠券覆盖核心场景
2. **门店为锚**：所有营销围绕"门店"展开，强化实体信任
3. **人群适配**：同一套系统，不同人群看到不同侧重
4. **模拟兼容**：充值/奖励用余额体系，后续可无缝对接真实支付

### 4.2 模块一：会员充值体系

#### 设计思路

**不做**：复杂的积分系统、等级权益矩阵（MVP 阶段过度设计）
**做**：余额充值 + 会员折扣 + 赠送优惠券（简单直接，用户能算清账）

#### 充值方案

| 档位 | 价格 | 渔具折扣 | 旅行立减 | 赠送 | 目标用户 |
|:---|:---|:---|:---|:---|:---|
| 体验卡 | ¥200 | 9.5折 | ¥10/次 | ¥10旅行券×1 | 新用户试水 |
| 畅玩卡 | ¥500 | 9折 | ¥20/次 | ¥20旅行券×2 + 装备免费租×1 | 钓鱼发烧友 |
| 尊享卡 | ¥1000 | 8.5折 | ¥30/次 | 免费出行×1 + 装备免费租×3 | 重度用户/退休 |

**赠送逻辑**：充值金额入余额，赠送券发到优惠券账户（非余额）

#### 与 Go 版差异

| 维度 | Go 版(0003) | cloud-mvp 建议 |
|:---|:---|:---|
| 余额支付 | 有 | 保留（核心） |
| 折扣计算 | 后端自动 | 保留（简化：下单时自动计算） |
| 充值记录 | 有 | 保留 |
| 会员等级展示 | 有 | 保留（简化为卡片式） |
| **新增**：赠送券 | 无 | 充值送优惠券（组合激励） |

#### 数据模型

```json
// users 集合扩展
{
  "balance": 500,           // 余额（分）
  "memberLevel": 2,         // 会员等级 0/1/2/3
  "memberExpireAt": null,   // 会员过期时间（暂不做）
  "totalRecharge": 1000     // 累计充值（用于等级判断）
}

// recharges 集合
{
  "_id": "自动生成",
  "openid": "用户openid",
  "amount": 500,            // 充值金额
  "giftAmount": 0,          // 赠送金额（入余额）
  "giftCoupons": ["券ID"],  // 赠送优惠券
  "level": 2,               // 充值后等级
  "status": "paid",         // paid/refunded
  "createdAt": "时间"
}
```

### 4.3 模块二：裂变获客体系

#### 设计思路

**不做**：复杂的分销层级、团长制度（MVP 阶段 + 合规风险）
**做**：老带新双向奖励 + 多人同行优惠（简单、可感知、无套路）

#### 裂变机制

**机制 A：邀请有礼（核心）**

| 角色 | 奖励 | 条件 |
|:---|:---|:---|
| 邀请人（老客） | ¥20 余额 | 被邀请人首次支付成功后发放 |
| 被邀请人（新客） | ¥15 首单立减 | 首次下单自动抵扣 |

**邀请码设计**：
- 每个用户有唯一 6 位邀请码
- 分享小程序页面自带邀请码参数
- 新用户通过邀请链接注册 → 自动绑定关系

**机制 B：多人同行优惠**

| 人数 | 优惠 | 说明 |
|:---|:---|:---|
| 2人同行 | 每人立减 ¥10 | 同一订单 |
| 3人+ | 每人立减 ¥15 + 赠鱼饵 | 同一订单 |

**机制 C：新人首单礼**

- 新注册用户自动获得 ¥10 旅行券
- 有效期 30 天

#### 与 Go 版差异

| 维度 | Go 版(0005) | cloud-mvp 建议 |
|:---|:---|:---|
| 邀请码 | 有 | 保留 |
| 绑定关系 | 有 | 保留 |
| 新客首单立减 | ¥15 | 保留 |
| 老客返现 | ¥20入余额 | 保留 |
| 邀请记录页 | 有 | 保留 |
| **新增**：同行优惠 | 无 | 多人同行阶梯优惠 |
| **新增**：新人券包 | 无 | 注册送券 |

#### 数据模型

```json
// users 集合扩展
{
  "inviteCode": "ABC123",    // 我的邀请码
  "invitedBy": "openid_xxx", // 谁邀请的我
  "isNewUser": true          // 新用户标记（用于首单立减）
}

// referrals 集合
{
  "_id": "自动生成",
  "inviterOpenid": "邀请人",
  "inviteeOpenid": "被邀请人",
  "inviteCode": "ABC123",
  "rewardAmount": 20,        // 邀请人奖励
  "newUserDiscount": 15,     // 新人立减
  "status": "pending",       // pending/completed
  "orderId": "首单订单ID",    // 触发奖励的订单
  "createdAt": "时间"
}
```

### 4.4 模块三：优惠券体系

#### 设计思路

**不做**：复杂的券类型矩阵、满减阶梯（MVP 阶段过度设计）
**做**：固定几种券类型，覆盖核心场景

#### 券类型设计

| 券类型 | 面额 | 使用条件 | 获取方式 | 有效期 |
|:---|:---|:---|:---|:---|
| 新人券 | ¥10 | 无门槛 | 注册自动发放 | 30天 |
| 旅行券 | ¥10/20/30 | 限旅行订单 | 充值赠送 | 90天 |
| 复购券 | ¥15 | 限旅行订单 | 完成出行后发放 | 30天 |
| 邀请券 | ¥15 | 限旅行订单 | 被邀请人首单 | 30天 |
| 唤醒券 | ¥20 | 限旅行订单 | 30天未消费推送 | 15天 |
| 装备租免券 | 免费 | 限租赁项 | 充值赠送/活动 | 60天 |

#### 优惠券数据模型

```json
// coupons 集合
{
  "_id": "自动生成",
  "openid": "用户openid",
  "type": "travel",          // travel/equipment/newuser/invite
  "amount": 15,              // 面额（分）
  "minAmount": 0,            // 最低消费（0=无门槛）
  "source": "recharge",      // recharge/referral/register/activity/system
  "status": "unused",        // unused/used/expired
  "orderId": "",             // 使用的订单ID
  "expireAt": "过期时间",
  "createdAt": "发放时间"
}
```

#### 优惠券使用流程

```
下单时：
1. 查询用户可用优惠券（status=unused & expireAt > now）
2. 按类型筛选适用券（旅行券→旅行订单，装备券→租赁订单）
3. 用户选择是否使用
4. 订单金额 = 原价 - 优惠券面额（优惠券金额不能超过订单金额）
5. 标记券为已使用
```

### 4.5 三模块联动关系

```
新用户注册
  │
  ├── 自动获得：新人券(¥10) + 邀请码
  │
  ├── 首单（使用新人券 + 邀请立减¥15）
  │     │
  │     └── 完成后：邀请人获¥20余额 + 用户获复购券(¥15)
  │
  ├── 复购（使用复购券）
  │     │
  │     └── 触发：充值引导（"充500本次省¥20"）
  │
  ├── 充值（获得余额 + 旅行券 + 折扣权益）
  │     │
  │     └── 后续下单自动：余额抵扣 + 会员折扣 + 优惠券叠加
  │
  └── 分享（邀请好友 → 双向奖励 → 持续裂变）
```

### 4.6 优惠叠加规则

| 场景 | 可叠加项 | 优先级 |
|:---|:---|:---|
| 普通用户下单 | 优惠券（1张） | 券 > 原价 |
| 会员用户下单 | 会员折扣 + 优惠券（1张） | 折扣先算，券后减 |
| 被邀请人首单 | 邀请立减 + 新人券 | 邀请立减优先，券叠加 |
| 多人同行 | 同行优惠 | 与会员折扣/券不叠加（取最优惠） |

**规则**：
- 每笔订单最多使用 1 张优惠券
- 会员折扣与优惠券可叠加（先折扣后用券）
- 多人同行优惠与券/折扣取最优惠（不叠加）
- 余额抵扣可与任何优惠叠加

### 5.1 新增/改造云函数

| 云函数 | action | 说明 |
|:---|:---|:---|
| `users` | `getProfile` | 已有，扩展返回余额/等级/邀请码 |
| `users` | `recharge` | **新增**：发起充值 |
| `users` | `useBalance` | **新增**：余额抵扣（下单时调用） |
| `referral` | `getInviteInfo` | **新增**：查询邀请码/邀请记录/奖励 |
| `referral` | `bindInvite` | **新增**：绑定邀请关系 |
| `referral` | `grantReward` | **新增**：发放邀请奖励（内部调用） |
| `coupons` | `list` | **新增**：查询我的优惠券 |
| `coupons` | `grant` | **新增**：发放优惠券（内部调用） |
| `coupons` | `use` | **新增**：使用优惠券（下单时调用） |
| `orders` | `create` | 改造：支持余额抵扣+优惠券+会员折扣+邀请立减 |

### 5.2 下单费用计算流程（改造）

```
orders.create 改造后：

1. 查询用户信息（余额/等级/邀请关系）
2. 查询可用优惠券
3. 计算基础费用 = 成人价×成人数 + 儿童价×儿童数
4. 应用会员折扣（如有）
5. 应用邀请立减（如被邀请人首单）
6. 应用同行优惠（如 3 人+）
7. 取最优：会员折扣 vs 同行优惠
8. 应用优惠券（用户选择的 1 张）
9. 应用余额抵扣（用户选择用多少余额）
10. 最终金额 = max(0, 折后价 - 券 - 余额)
11. 创建订单，标记使用的券和余额
```

## 八、风险与对策

| 风险 | 影响 | 对策 |
|:---|:---|:---|
| 无商户号，充值不能走真实支付 | 用户无法真正充值 | MVP 用模拟充值（后台手动加余额），后续对接支付 |
| 优惠券刷单 | 薅羊毛 | 限制每人每种券数量，邀请关系唯一绑定 |
| 优惠叠加计算复杂 | 用户困惑 | 前端清晰展示分项，取最优惠逻辑自动化 |
| 退休用户不会操作 | 裂变效果差 | 门店店员辅助操作，线下引导 |

## 九、与 Go 版(0003/0005)对比总结
